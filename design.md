# AI 自动答题工具 — 产品设计文档

> 版本：v0.1 草稿
> 日期：2026-03-31

---

## 一、项目概述

一款面向企业培训场景的桌面端 AI 辅助答题工具。核心思路是：**题库优先，AI 兜底**。答题时优先从本地题库模糊匹配答案，匹配不到时再调用 AI 推理，同时支持半自动（仅显示答案）和全自动（自动点击）两种工作模式。

**目标用户：** 个人及小范围朋友使用
**开发语言：** Python
**运行平台：** Windows（默认主屏幕）

---

## 二、核心设计原则

1. **题库优先** — 本地 SQLite 题库模糊匹配，命中则直接返回，不消耗 AI 调用
2. **AI 兜底** — 题库未命中时，调用 AI 推理答案
3. **识别双轨** — OCR 文字识别 + AI 视觉识别并行，由 AI 判断采用哪路结果
4. **缓存去重** — 相同截图/题目不重复调用 AI，直接复用上次结果
5. **非侵入** — 不注入浏览器，不修改页面，仅通过截图 + 鼠标模拟操作

## 三、功能模块

### 3.1 主界面（控制窗口）

普通桌面窗口，用于启动前的配置操作：

```
┌─────────────────────────────────────┐
│  AI 自动答题工具                      │
├─────────────────────────────────────┤
│  题库：[下拉选择]  [导入] [题库管理]  │
│  模式：○ 半自动（仅显示答案）          │
│        ○ 全自动（自动点击）            │
│                                     │
│  AI 配置：[设置]                      │
│                                     │
│  [  启  动  ]   [  停  止  ]         │
└─────────────────────────────────────┘
```

> 启动后控制窗口可最小化，所有运行状态和答案均显示在悬浮 HUD 条上。
> "题库管理"按钮打开独立弹窗，支持导入 Excel 和分页预览当前题库内容。

### 3.1.1 悬浮 HUD 状态条

程序启动后在屏幕**顶部居中**出现的常驻悬浮条，用于显示运行状态、题目和答案。

**交互特性：**
- 无边框、无标题栏
- 始终置顶（`-topmost`），不被浏览器遮挡
- 鼠标穿透（`WS_EX_TRANSPARENT`），点击直接穿透到下层窗口，不影响答题操作
- 不可拖动、不可关闭（只能通过主窗口停止按钮关闭）

**外观：**
- 浅灰底色 + 半透明，透明度在设置中可调（默认 85%）
- 宽度固定为屏幕宽度的 50%，水平居中
- 距屏幕顶部距离在设置中可调（默认 20px）

**显示内容（单行紧凑布局）：**

```
[状态] 题目：xxxxxxxxx...  |  来源：题库  |  答案：A. 正确答案文字
```

**题目省略规则：**
- 计算当前 HUD 宽度能容纳的最大字符数
- 若题干超出可用空间，保留前 20% 字符 + `...` 省略后 80%
- 答案和来源字段不省略，优先保证答案可见

**状态文字对照：**

| 状态 | 显示文字 |
|------|----------|
| 程序已启动，等待截图 | `⏳ 等待中` |
| 正在识别题目 | `🔍 识别中...` |
| 题库命中 | `✅ 题库  题目：xxx  答案：xxx` |
| AI 推理中 | `🤖 AI 推理中...` |
| AI 命中 | `✅ AI  题目：xxx  答案：xxx` |
| 已作答跳过 | `⏭ 已答过，跳过` |
| 识别失败 | `❌ 识别失败，请手动作答` |
| AI 超时/失败 | `❌ AI 失败，请手动作答` |
| 点击失败 | `❌ 点击失败，请手动选择` |
| 已选择完毕 | `✔ 已选择，请手动提交` |
| OCR 模型缺失 | `⚠ 未检测到 OCR 模型，仅使用 AI Vision 识别` |

> **半自动模式说明：** HUD 显示题目和答案后，立即将该题标记为已作答（answered=1），用户自行在浏览器中点击对应选项，程序不做任何点击操作。后续定时截图若 pHash 未变化，直接跳过；若画面变化（新题），继续处理新题。

### 3.2 题库管理模块

- **导入：** 从 Excel 文件导入，每个 Excel 文件对应一个独立的 `.db` 数据库文件，文件名与 Excel 同名
- **Excel 格式约定：** 固定两列，A 列为题干，B 列为答案（第一行为表头，从第二行开始读取）
- **存储：** SQLite3 本地数据库，多题库并存，主界面下拉切换
- **功能范围：** 仅支持导入和列表预览，不支持在工具内编辑或删除题目（直接编辑 Excel 后重新导入）
- **列表预览：** 分页显示，每页 50 条，避免数据量大时卡顿
- **字段结构：**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER | 主键 |
| question | TEXT | 题目原文 |
| answer | TEXT | 正确答案；多选题用 `\|答案分隔\|` 分隔，如 `A\|答案分隔\|B\|答案分隔\|C`；填空多空同理 |
| created_at | DATETIME | 录入时间 |

> 题库只存题干和答案，选项内容由 AI 从截图中识别，不需要预存。

- **模糊搜索算法：** 使用 Python 内置 `difflib.SequenceMatcher` 计算题干相似度，相似度 ≥ 阈值（默认 0.8）则命中，返回相似度最高的一条记录

### 3.3 截图与识别模块

**触发方式：** 用户点击启动后，工具按设定间隔自动定时轮询截图，半自动和全自动模式均为定时轮询，无需手动触发。截图间隔在设置中配置（默认 2s）。

**鼠标排除处理：** 截图后，用 `pyautogui.position()` 获取当前鼠标坐标，将鼠标周围 32×32 像素区域涂黑，再计算 pHash，确保同一道题无论鼠标位置如何 hash 值一致。

**截图去重（pHash）：**
- 使用感知哈希（pHash，`imagehash` 库）而非 MD5
- 两张截图 pHash 汉明距离 ≤ 8 则视为同一画面，直接跳过本轮
- 避免光标闪烁、细微动画导致误判为新题

**双轨识别流程：**

```
全屏截图 → 涂黑鼠标区域 → 计算 pHash
    │
    ├── pHash 命中缓存且已作答 ──► 跳过本轮
    │
    ├──► OCR 识别（本地，PaddleOCR）──► 文字结果 A
    │
    └──► 截图直接发给 AI Vision ──────► 文字结果 B
                                            │
                                    AI 判断采用哪路结果
                                    （综合置信度、完整性）
                                            │
                                    最终题目文字
```

**判断逻辑（由 AI 执行）：**
- OCR 结果乱码率高 → 采用 Vision 结果
- 题目含图表/特殊排版 → 采用 Vision 结果
- OCR 结果清晰完整 → 采用 OCR 结果（节省 token）

**错误处理：**
- OCR 失败 + AI Vision 也失败 → HUD 显示"❌ 识别失败，请手动作答"，本轮跳过，继续下次截图循环

      │

  ┌─────────────────────────────────────────┐
  │ 半自动                   │ 全自动        │

### 3.5 缓存模块

缓存存储在程序目录的 `cache.db`（SQLite），同时在内存中保留运行期副本加速查询。

**cache.db 表结构：**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER | 主键 |
| question_hash | TEXT | 题目文字 MD5 hash，唯一索引 |
| phash | TEXT | 截图感知哈希（pHash 字符串） |
| answer | TEXT | 答案内容 |
| source | TEXT | 来源：`db`（题库）或 `ai` |
| answered | INTEGER | 是否已作答：0 未答 / 1 已答 |
| created_at | DATETIME | 首次写入时间 |
| updated_at | DATETIME | 最后更新时间 |

**查询逻辑：**
- 先用截图 pHash 做汉明距离比较（≤ 8 视为同一画面）
- pHash 命中且 `answered=1` → 跳过本轮（包括半自动模式下已显示过答案的情况，不重复处理）
- pHash 未命中，用题目文字 MD5 查 `question_hash`
  - 命中 → 复用答案，重新执行点击/显示（answered 仍为 0）
  - 未命中 → 走题库搜索，题库无匹配则直接让 AI 自行推理，写入新记录

**过期清理：** 启动时自动删除 `created_at` 超过配置天数（默认 7 天）的记录

### 3.6 鼠标自动点击模块（全自动模式）

**职责范围：仅负责选项点击，不点击"提交"和"下一题"按钮，由用户自行操作。**

**流程：**
1. AI 识别出题目和选项后，同时返回各选项在截图中的坐标区域
2. 根据答案确定目标坐标
3. 调用 `pyautogui` 移动鼠标并点击对应选项
4. 点击后单独截图发给 AI 确认选中状态，结果更新到 HUD

**坐标计算：**
- AI Vision 返回选项的相对坐标（百分比）
- 工具根据当前屏幕分辨率换算为绝对像素坐标
- 点击前加入随机微小偏移（模拟人工操作，避免过于机械）

**支持题型的点击逻辑：**

| 题型 | 点击行为 |
|------|----------|
| 单选 | 点击对应选项 |
| 多选 | 依次点击所有正确选项 |
| 判断 | 点击"对"或"错" |
| 填空 | 定位输入框，模拟键盘输入 |
| 简答 | 定位文本域，模拟键盘输入 |

### 3.8 AI Prompt 设计

所有 prompt 均要求 AI 返回 JSON，便于程序解析。

---

**Prompt A — 题目识别（双轨结果 + 坐标）**

```
你是一个答题辅助系统，负责从截图中识别题目信息。

我会同时提供：
1. OCR 识别的文字结果
2. 原始截图

OCR 文字结果：
{ocr_text}

请分析截图，完成以下任务：
1. 判断 OCR 结果是否准确（乱码多、排版混乱、含图表则优先用截图）
2. 识别题目类型：single（单选）/ multi（多选）/ judge（判断）/ fill（填空）/ essay（简答）
3. 提取题干文字
4. 提取所有选项文字及其在截图中的中心坐标（相对截图宽高的百分比，0~1）
5. 如有字数要求，提取字数限制

返回严格 JSON，不要有任何多余文字：
{
  "source": "ocr" 或 "vision",
  "type": "single/multi/judge/fill/essay",
  "question": "题干文字",
  "options": [
    {"label": "A", "text": "选项文字", "x": 0.25, "y": 0.45},
    {"label": "B", "text": "选项文字", "x": 0.25, "y": 0.52}
  ],
  "word_limit": null 或 数字
}
```

---

**Prompt B — 答案推理（题库无匹配时调用）**

```
你是一个企业培训答题助手，请根据题目内容直接给出正确答案。

题目类型：{type}
题干：{question}
选项：{options_text}
{word_limit_hint}

请直接给出答案，不要解释过程。

返回严格 JSON，answer 字段规则如下：
- 单选题/判断题：字符串，如 "A" 或 "正确" 或 "错误"
- 多选题：用 |答案分隔| 连接，如 "A|答案分隔|B"
- 填空题/简答题：直接填写文字内容

{
  "answer": "此处填写答案",
  "confidence": 0.9
}
```

> `word_limit_hint`：若有字数要求则为 `"字数要求：不少于 {n} 字"`，否则为空字符串。
> 简答题无字数要求时，AI 自行判断合适篇幅。

---

**Prompt C — 点击确认**

```
请观察这张截图，判断以下选项是否已被选中（高亮、勾选、填充等视觉变化均算选中）。

需要确认的选项：{expected_labels}

返回严格 JSON：
{
  "confirmed": true 或 false,
  "detail": {"A": true, "B": false}  // 每个选项的选中状态
}
```

### 3.9 配置文件初始化

**config.json 自动创建规则：**
- 程序启动时检查运行目录是否存在 `config.json`
- 不存在则自动创建，写入所有字段的默认值（API Key 等敏感字段留空）
- 存在但字段缺失时，自动补全缺失字段为默认值，不覆盖已有配置

**config.json 默认结构：**
```json
{
  "provider": "openai",
  "api_key": "",
  "api_base_url": "https://api.openai.com/v1",
  "model": "",
  "timeout": 30,
  "similarity_threshold": 0.8,
  "cache_expire_days": 7,
  "screenshot_interval": 2,
  "hud_opacity": 0.85,
  "hud_top_offset": 20
}
```

**启动按钮状态逻辑：**

| 配置状态 | 启动按钮显示 | 是否可点击 |
|----------|-------------|-----------|
| api_key 为空 | `请先完善配置` | 否（灰色禁用）|
| model 为空 | `请先完善配置` | 否（灰色禁用）|
| 未选择题库 | `请先选择题库` | 否（灰色禁用）|
| 配置完整 | `启 动` | 是 |

> 点击灰色按钮无响应，不弹窗，用户看到文字提示即可知道原因。

## 五、数据流总览

```
用户启动
   │
   ▼
检测 OCR 模型 → 无模型：单轨（仅 AI Vision）/ 有模型：双轨
   │
   ▼
定时轮询截图（全屏，涂黑鼠标区域）
   │
   ▼
计算 pHash → 与上次比较汉明距离
   │
   ├── 距离 ≤ 8 且 answered=1 ──► 跳过本轮
   │
   ├── 距离 ≤ 8 且 answered=0 ──► 已有答案，直接执行点击/显示
   │
   └── 距离 > 8（新画面）
           │
           ▼
       OCR 识别 + AI Vision 识别（并行）
           │
       AI 判断采用哪路结果 → 最终题目文字
           │
       题目文字 MD5 查缓存
           │
       ├── 命中且 answered=1 → 跳过
       │
       └── 未命中 → 题库模糊搜索
                   │
               命中题库 → 写缓存 → 执行
                   │
               未命中 → AI 推理（带超时）→ 写缓存 → 执行
                                │
                          ┌─────┴─────┐
                       半自动        全自动
                    HUD 显示答案   鼠标自动点击
                    标记 answered=1  截图确认 → 标记 answered=1
```

## 六、技术栈

| 模块 | 技术选型 | 说明 |
|------|----------|------|
| UI 框架 | tkinter（内置） | 无需安装，打包体积最小，界面简洁够用 |
| OCR | PaddleOCR（百度飞桨） | 模型文件放运行目录，程序自动检测并选择可用模型 |
| 截图 | mss + Pillow | mss 高性能截图，Pillow 处理图像 |
| 图像 hash | imagehash | pHash 感知哈希，用于截图去重 |
| 鼠标控制 | pyautogui | 鼠标移动与点击 |
| 数据库 | SQLite3（内置） | 题库 + 缓存，无需安装数据库服务 |
| AI 调用 | openai SDK | 兼容 OpenAI / Claude / 本地模型等所有兼容接口 |
| Excel 导入 | openpyxl | 读取 .xlsx 文件 |
| 配置存储 | JSON 文件 | 存放 API Key 等配置，Key 本地加密 |
| 通知提醒 | tkinter 无边框置顶窗口（HUD） | 自绘，无需额外库，鼠标穿透靠 pywin32 |
| pywin32 | win32gui / win32con | 设置窗口鼠标穿透样式（WS_EX_TRANSPARENT）|
| 打包 | PyInstaller | 生成单文件 .exe，目标体积 ≤ 30MB |

**开发环境搭建（极简）：**
1. 安装 Python 3.11（官网下载，勾选 Add to PATH）
2. 运行 `pip install mss pillow imagehash pyautogui openai openpyxl paddlepaddle paddleocr pywin32 pyinstaller`
3. 完成，无需其他配置

> PaddleOCR 模型不随程序打包，用户自行下载放到运行目录 `models/` 下。无模型时程序仍可运行，仅使用 AI Vision 单轨识别。

## 七、目录结构（规划）

```
AIAutoAnswering/          ← 程序运行目录（.exe 所在位置）
├── AIAutoAnswering.exe   # 打包后的可执行文件
├── config.json           # 用户配置（AI Key、超时等）
├── db/                   # 题库目录，程序启动时扫描此目录下所有 .db 文件
│   ├── 培训题库A.db
│   └── 培训题库B.db
├── cache.db              # 答题缓存（pHash + 题目 hash + 已答记录）
├── models/               # PaddleOCR 模型目录（用户自行下载放置）
│   ├── det/              # 文字检测模型（轻量版 or 标准版）
│   ├── rec/              # 文字识别模型（轻量版 or 标准版）
│   └── cls/              # 方向分类模型
└── assets/
    └── icon.ico
```

**PaddleOCR 模型选择规则：**
- 程序启动时扫描 `models/` 目录
- 若检测到标准模型（server 版）→ 优先使用标准模型
- 若只有轻量模型（mobile 版）→ 使用轻量模型
- 若 `models/` 目录为空或不存在 → HUD 显示"⚠ 未检测到 OCR 模型，仅使用 AI Vision 识别"，跳过 OCR 轨道，单轨运行
- 用户可在程序设置界面查看当前使用的模型版本及下载说明链接

**题库文件管理规则：**
- 所有 `.db` 文件统一放在程序目录的 `db/` 子目录下
- 主界面下拉列表在启动时自动扫描 `db/` 目录，列出所有可用题库
- 导入 Excel 时，在 `db/` 目录下生成同名 `.db` 文件（如 `培训题库A.xlsx` → `db/培训题库A.db`）
- **题库必须在点击启动前选择**，启动后不允许切换（避免运行中状态混乱）

## 八、已确认事项 & 待讨论

**已确认：**
- [x] OCR 使用 PaddleOCR，模型放运行目录 models/，用户自行下载；有标准模型优先用标准，无模型则跳过 OCR 单轨运行
- [x] 题库管理只支持导入和分页列表预览（每页50条），不支持工具内编辑删除
- [x] 题库无匹配时直接调用 Prompt B 让 AI 自行推理，不传题库答案
- [x] 半自动模式下 pHash 命中且已显示过答案（answered=1）直接跳过，不重复处理，用户停留在同一题不影响
- [x] Prompt 设计：识别题目（返回题型/题干/选项坐标）、推理答案、确认点击结果，均返回 JSON
- [x] 多选题答案用 `|答案分隔|` 分隔，填空多空同理
- [x] config.json 不存在时自动创建，字段缺失时自动补全，不覆盖已有配置
- [x] 配置不完整或未选题库时，启动按钮显示提示文字并禁用，不弹窗
- [x] 每个 Excel 对应独立 `.db` 文件，存放在程序目录 `db/` 下，主界面下拉切换
- [x] 题库必须在点击启动前选择，启动后不可切换
- [x] 全自动模式只点击选项，不点提交/下一题，由用户手动操作
- [x] 点击选项后单独截图发给 AI 确认是否选中成功
- [x] 完成选择/失败等所有状态通过屏幕顶部居中悬浮 HUD 条显示，不用托盘通知
- [x] HUD 条无边框、置顶、鼠标穿透（点不到），浅灰底色半透明，透明度和顶部距离可在设置中调整
- [x] 所有失败情况（识别失败、AI 超时、点击失败）均通过 HUD 显示，不弹窗
- [x] 答题记录保存在缓存中，同时作为"已答题目"去重依据，重复截图到同一题时自动跳过
- [x] 截图去鼠标区域后用 pHash 做图像级去重，防止光标闪烁误判
- [x] 简答题字数：题目有要求则遵守，无要求则 AI 自行判断篇幅；风格无约束
- [x] UI 框架选用 tkinter（内置，打包体积最小，开发最简单）
- [x] 程序运行目录即为数据目录，`.db` 和 `config.json` 均在程序旁边

**待讨论：** 无，需求已全部确认，可进入开发阶段。
